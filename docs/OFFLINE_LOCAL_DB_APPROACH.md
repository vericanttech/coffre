# Offline local database — approach (parallel to Firestore)

This document describes the **right approach** to giving users offline access to their document list and metadata (search, info) without relying on the network, using a **local database in parallel to Firestore**. It does not contain code; it explains the architecture and how the pieces fit together.

---

## 1. What you have today

**Service worker (Workbox)**  
- **Role:** Caches the **app shell** (JS, CSS, HTML from the build) and serves `index.html` for navigation requests.  
- **Effect:** When the user has no internet, the app still **loads** (blank or shell).  
- **What it does not do:** It does **not** cache the document list, thumbnails, or any Firestore data.  
- **Where it lives:** Custom logic in `src/sw.ts`; the built `dist/sw.js` is generated by the Vite PWA plugin and includes Workbox precaching + a navigation fallback.

**Data today**  
- Document list and metadata come **only from Firestore** via `streamDocuments` (real-time listener).  
- When the device is offline, Firestore cannot deliver data, so the list is empty and search has nothing to work on.  
- Thumbnails and originals are loaded from Firebase Storage via URLs; those requests also fail offline unless something has cached them.

So: **offline today = app shell only**. No document list, no search, no images.

---

## 2. Goal

- **When the user has no internet:** They can still open the app and:
  - See their **document list** (same as when online, or a recent snapshot).
  - **Search** through that list (by title, category, summary, keywords) and see results.
  - Open a **document detail** and see all **metadata** (title, category, dates, extracted fields, etc.).
  - Optionally see **thumbnails** or **images** only for documents that were already cached (e.g. previously viewed). If a picture was never cached, it is acceptable to show a placeholder and only the metadata.

- **When the user is back online:** The app should stay in sync with Firestore (new/updated/deleted docs) and optionally refill or refresh the local copy.

So the local store is a **parallel copy** of the document list + metadata, used when the network is unavailable; it is **not** a replacement for Firestore as the source of truth when online.

---

## 3. Why a local database (and not only the service worker)

**Service worker (SW)**  
- Good at: caching **HTTP responses** (static assets, or API responses as opaque blobs).  
- Limitations:  
  - It does not “query” by your app’s logic (e.g. “all documents for user X”, “search by keyword”). You would have to cache whole list responses and serve them back.  
  - It has no notion of “document list” vs “thumbnails” vs “metadata” as separate queryable structures.  
  - Organizing search (e.g. full-text or filtered) and keeping a mirror of Firestore’s structure is easier in an **application-level** store that the main app (and optionally the SW) can read and write in a structured way.

**Local DB (e.g. IndexedDB)**  
- Good at: storing **structured data** (list of documents, one record per doc with title, category, keywords, etc.), **queryable** by the app (e.g. “give me all docs for user U”, “filter by keyword”).  
- The **same** document list you show in the UI can be written to IndexedDB when you receive it from Firestore; when offline, the UI reads from IndexedDB instead of Firestore.  
- Thumbnails (or full images) can be stored either in the same DB as blobs or in a separate cache (e.g. Cache API used by the SW, or a second IndexedDB store). The important point: **document list + metadata** are what enable offline list and search; images are an optional layer on top.

So the **right split** is:

- **Local DB (IndexedDB, app-level):** Mirror of **document list + metadata** for the current user. Used for offline list, search, and detail metadata. Optional: store thumb (or original) blobs here or in a dedicated cache.
- **Service worker:** Continues to handle **app shell** (precache). Optionally, it can also cache **thumbnail/original URLs** (e.g. runtime cache) so that when the user opens a doc offline, the SW can serve the image from cache if it was previously loaded. The SW does **not** need to know about “document list” or “search”; that stays in the app and the local DB.

---

## 4. Recommended architecture in short

1. **Single source of truth when online:** Firestore (and Storage for files). The app keeps using `streamDocuments` and Firestore as today.

2. **Local mirror (IndexedDB):**  
   - One store (or table) for **vault documents**: same shape as your app’s document type (id, userId, title, category, keywords, ocrSummary, extractedFields, thumbRef, originalRef, etc.).  
   - Key by `userId` and `docId` (or a composite) so you can read “all docs for user U” and update/delete by doc id.  
   - When the app receives data from Firestore (stream or initial load), it **writes** that set of documents into IndexedDB (upsert per doc, remove docs that no longer exist if you want a full mirror).  
   - When the app detects **offline** (or Firestore is unavailable), it **reads** the document list from IndexedDB instead of Firestore and runs **search/filter in memory** on that list (same as today: filter by query on title, category, summary, keywords).  
   - Document detail screen: metadata comes from the same local list (or a single-doc read from IndexedDB). Images: if you have a cached copy (e.g. via SW or a blob in IndexedDB), show it; otherwise show placeholder and only metadata.

3. **Service worker:**  
   - Keep **precache** of the app shell and navigation fallback.  
   - Optionally add a **runtime cache** for Firebase Storage URLs (thumb and/or original): when the app requests a thumb or image URL, the SW fetches it and caches the response; next time (e.g. offline) it can serve from cache. That way “picture not available” offline only happens for docs the user never opened (or whose thumb was never requested).  
   - The SW does **not** need to write or read the document list; that stays in the main app and IndexedDB.

4. **Sync and freshness:**  
   - When **online:** Firestore stream updates the UI and, in the same flow, updates IndexedDB. So the local DB is always a reflection of what Firestore last sent.  
   - When **offline:** No sync; the app only reads from IndexedDB.  
   - When **back online:** Resume the Firestore stream; new updates overwrite the local mirror again. Optionally, you can implement “queue writes when offline and flush when online” for uploads or edits; that is a separate concern from “local DB for read-only offline list/search”.

5. **Auth:**  
   - IndexedDB is per-origin. You should store documents **per userId** (e.g. key by userId or a “current user” id). When the user logs out, you can clear or scope the local DB to avoid leaking another user’s data if they log in on the same device. When they log in, you load from Firestore and repopulate the mirror for that user.

---

## 5. What goes where

| What | Where | Purpose |
|------|--------|--------|
| App shell (JS, CSS, index.html) | Service worker (precache) | App loads offline. |
| Document list + metadata | IndexedDB (app) | Offline list, search, document detail metadata. |
| Thumbnails / originals | Optional: Cache API (SW) or IndexedDB blobs | Show images offline for already-seen docs. |
| Firestore | Server | Source of truth when online; stream updates and mirror to IndexedDB. |

The **critical** part for “search and get info without the picture” is: **document list + metadata in IndexedDB**. Once that is in place, the rest (images) is an improvement on top.

---

## 6. Order of implementation (conceptual)

1. **Introduce a local DB (IndexedDB)** for vault documents: open a DB, create a store with the document shape, and define “write list of docs for user” and “read list of docs for user”.
2. **Data layer:** When online and you receive documents from Firestore (e.g. in the same place you currently feed `streamDocuments` into state), also write those documents to IndexedDB (per user). When you detect offline (or Firestore fails), read the list from IndexedDB and use that for the UI instead.
3. **Search:** Your existing search logic (filter by query on title, category, summary, keywords) already runs on an in-memory list. That list will now come either from Firestore (online) or from IndexedDB (offline). No change to the search algorithm; only the **source** of the list changes.
4. **Document detail:** When opening a doc, metadata can come from the same list (or a single read from IndexedDB by id). Image: try to load from cache (e.g. SW cache or blob in DB); if missing, show placeholder and only metadata.
5. **Service worker (optional):** Add runtime caching for Storage URLs so that thumbnails (and optionally originals) that were ever loaded get served from cache when offline.

---

## 7. Summary

- **Local DB parallel to Firestore** = store a mirror of the **document list + metadata** in IndexedDB (app-level), updated whenever Firestore data is received, and read when offline so the user can still browse and search. Images are optional (cache when possible; otherwise “picture not available” with full metadata is acceptable).
- **Service worker** in your app today = app shell only. Keep it that way for shell; optionally use it to cache thumbnail/original responses. The SW does not need to implement the “local DB” or search; that is the app’s job using IndexedDB.
- This gives you: **offline list, offline search, offline metadata**, and optionally offline images for already-cached items, without writing code inside the SW for document list or search.
